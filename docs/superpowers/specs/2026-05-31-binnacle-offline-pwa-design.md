# Binnacle Differentiator: Offline and PWA Caching Design

Status: approved. This is the offline-first pipeline named in the foundation plan as "the spec
immediately after the foundation." It follows the foundation design and the rules in `CLAUDE.md`:
offline-first, no app-code CDN, bundle the app's own assets, 100% Signal K conformance, and the
Pi memory budget.

## 1. Purpose

Make Binnacle work with no internet. A bluewater cruiser loses connectivity offshore, so the chart
plotter must keep rendering the map and charts it has already shown, and the app itself must load
and run from the boat's Signal K server with no outside network. This spec adds three caching
layers: an installable PWA app shell, a runtime cache of the online base-map assets, and a cache of
the Signal K PMTiles chart range requests. The result: anywhere the navigator has already viewed
renders offline, and the app installs to the helm display's home screen and launches like a native
app.

This is the "heavy caching" the operator asked for. It also durably resolves the PMTiles
`ERR_CACHE_WRITE_FAILURE` (the no-store quick fix already stops the crash; the service worker now
serves those ranges from CacheStorage instead).

## 2. Decisions locked in this brainstorm

1. **Cache everything for offline.** The app shell (installable PWA), the OpenFreeMap base-map
   assets (style, vector tiles, glyphs, sprites, and the natural-earth raster), and the Signal K
   PMTiles chart range requests.
2. **View-cache now, prefetch later.** v1 caches assets as the navigator pans over them
   (cache-first runtime caching). A deliberate "download this region for the passage" control is a
   later refinement, not v1.

## 3. Architecture

The PWA is delivered by a service worker plus a web-app manifest, generated by `vite-plugin-pwa`
(which wraps Workbox). The service worker is a new artifact at the app boundary; it sits outside
the Feature-Sliced layers because it is a platform shell, not application logic. The app's
registration code is a thin `shared/pwa` module.

### Build: vite-plugin-pwa

- Add `vite-plugin-pwa` to `vite.config.ts` with `registerType: 'autoUpdate'` and Workbox runtime
  caching rules (below). Precaching covers the built app shell (the hashed JS and CSS, the HTML,
  the fonts, and the icons that Vite emits into `public/`), so the app loads offline.
- The manifest declares the app identity: name "Binnacle", short name, theme and background colors
  (from the day theme tokens), display `standalone`, the existing app icon, and the start URL
  `/binnacle/`. The same `signalk` manifest screenshots can feed the PWA manifest screenshots.
- Set the Workbox `navigateFallback` to the app's `index.html` so a deep launch still boots the
  shell offline.

### Runtime caching rules (Workbox)

1. **OpenFreeMap base map** (`https://tiles.openfreemap.org/*`): `CacheFirst`. Style JSON, vector
   `pbf` tiles, glyphs (fonts), sprites, and natural-earth raster. OpenFreeMap sends
   `access-control-allow-origin: *` and a one-day `cache-control`, so cross-origin runtime caching
   works. Cap the cache by entry count and age so it does not grow without bound; the cache holds
   what the navigator has viewed.
2. **Signal K PMTiles** (`/signalk/pmtiles/*`): `CacheFirst` with range-request support. PMTiles
   reads an archive through many HTTP range requests; Workbox's range-requests plugin serves cached
   ranges and satisfies `206 Partial Content` from the cache, which is what makes the 4.5 GB
   archive usable offline once viewed. The cache key ignores the `no-store` directive the app sets
   on the fetch, because the service worker, not the HTTP cache, is now the durable store.
3. **Signal K chart raster tiles and other `/signalk/...` resources the map requests**:
   `StaleWhileRevalidate` or `CacheFirst` as appropriate, so raster charts also cache.
4. **The Signal K data stream and REST API** (`/signalk/v1/stream`, `/signalk/v*/api/*`): NOT
   cached. Live navigation data must never be served stale, and the WebSocket is not a cacheable
   request anyway. The runtime rules explicitly exclude these paths.

### App integration: shared/pwa

- `shared/pwa/register.ts`: registers the service worker via `virtual:pwa-register` (the
  vite-plugin-pwa virtual module), with an update prompt hook. Imported once from the app entry.
- A small "new version available, reload" affordance (an update toast) when `autoUpdate` finds a
  new build, so a helm display picks up releases without a manual hard refresh.
- An offline indicator in the status strip (a derived `navigator.onLine` signal) so the navigator
  knows when the plotter is running cached, not live.

### The PMTiles no-store source and the service worker

The no-store quick fix and the service worker coexist correctly. The app's `NoStoreSource` sets
`cache: 'no-store'` to bypass the browser's broken HTTP disk cache. The service worker intercepts
that fetch before it reaches the network and serves from CacheStorage when present, so the durable
cache is the service worker's, and Chrome's failing disk cache is never involved. On a cache miss
the worker fetches from the network and stores the response.

## 4. Data flow

1. First load with internet: the service worker precaches the app shell; runtime rules cache the
   base-map assets and PMTiles ranges as the map requests them.
2. The navigator pans and zooms over the cruising area: every base tile, glyph, sprite, and PMTiles
   range fetched is stored.
3. Connectivity drops: the app loads from the precache, the map renders the cached base and chart,
   and the status strip shows an offline indicator. The Signal K stream reconnects when a local
   connection returns (the live data is same-origin on the boat's LAN, independent of internet).
4. A new build is published: `autoUpdate` fetches it in the background and the update toast offers a
   reload.

## 5. Signal K conformance

- The app remains a Signal K webapp served from `public/` at `/binnacle/`; the service worker and
  manifest ship inside that served directory and the `files` tarball.
- The live data paths (`/signalk/v1/stream`, `/signalk/v*/api/*`) are explicitly excluded from
  caching, so navigation data is never stale.
- The PMTiles and chart resources discovered through `/resources/charts` are cached by URL, with no
  assumption about a specific chart, honoring the generic chart adapter.
- Offline-first is realized: zero outside-network dependency for the app shell and for already
  viewed map and chart area.

## 6. Modularity

The service worker config lives in `vite.config.ts` (build concern); the registration and the
update and offline UI live in a `shared/pwa` module plus a tiny status-strip indicator. No
application feature depends on the PWA layer, and removing the plugin reverts to the non-installable
online-only app. dependency-cruiser stays green.

## 7. Build order (each a major step: ends with /cleanup and the gate)

1. Add `vite-plugin-pwa`, the manifest, and app-shell precaching; register the service worker from
   the entry. Verify the app installs and loads offline (shell only).
2. Add the OpenFreeMap runtime caching rule; verify the base map renders offline after a first
   online view.
3. Add the PMTiles range-request caching rule; verify a viewed chart area renders offline.
4. Add the update toast and the offline indicator in the status strip.

Each step is independently shippable and verified live in the browser (DevTools offline mode).

## 8. Deferred (own later specs)

A deliberate "download this region" prefetch with progress and storage-quota management, cache
eviction policy tuning, background sync, and push notifications. v1 is view-cache plus the
installable shell, which covers the common offshore case: you have looked at where you are going.

## 9. Open questions for review

- Cache size caps: what entry-count and age limits for the base-map and PMTiles caches suit the
  display device's storage? (Proposed: generous defaults, for example a few thousand base entries
  and an age cap of 30 days, tuned after measuring on the real display.)
- Whether the offline indicator should also surface the Signal K connection phase (already shown as
  "Connected"/"Reconnecting") or stay a separate internet-reachability signal. (Proposed: keep them
  separate; the Signal K connection is the boat LAN, internet reachability is the base-map source.)
